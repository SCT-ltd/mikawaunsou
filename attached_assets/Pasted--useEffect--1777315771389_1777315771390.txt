確認ポイント：

1. Selectのvalueとoption valueの型を統一してください。
もし option value が string なのに row.allowanceDefinitionId が number だと一致しない可能性があります。
Selectのvalueは必ず String() で統一してください。

例：
<Select
  value={row.allowanceDefinitionId ? String(row.allowanceDefinitionId) : ""}
  onValueChange={(value) => {
    const selected = allowanceDefinitions.find(
      def => String(def.id) === String(value)
    );

    updateAllowanceRow(row.id, {
      allowanceDefinitionId: selected.id,
      allowanceName: selected.name,
      name: selected.name,
      isTaxable: selected.isTaxable,
      taxable: selected.isTaxable,
    });
  }}
>

<SelectItem value={String(def.id)}>
  {def.name}
</SelectItem>

2. フィールド名の不一致をすべて洗い出してください。
以下のような名前が混在している可能性があります。

- allowanceDefinitionId
- allowance_definition_id
- allowanceId
- allowance_id
- allowanceTypeId
- allowance_type_id
- definitionId
- name
- allowanceName
- allowance_name
- isTaxable
- is_taxable
- taxable

保存、取得、row state、Select valueで使っている名称がバラバラだと消えます。
必ず1つの正規フィールドに統一してください。

フロントのrow stateでは以下に統一してください。

{
  id: string;
  allowanceDefinitionId: string;
  allowanceName: string;
  amount: number;
  isTaxable: boolean;
  sortOrder: number;
}

API/DBではsnake_caseでも構いませんが、変換箇所を明確にしてください。

DB:

フロント:

3. DB保存時に allowance_definition_id が null になっていないか確認してください。
もしDBに金額だけ保存され、allowance_definition_id が null の場合、再編集時にSelectは復元できません。
この場合は保存payload作成処理を修正してください。

4. GET APIで allowance_definition_id を返しているか確認してください。
DBに保存されていても、GETで返していなければフロントでは復元できません。
GETレスポンスには最低限以下を含めてください。

{
  id,
  allowanceDefinitionId,
  allowanceName,
  amount,
  isTaxable,
  sortOrder
}

またはAPIがsnake_caseを返すなら、フロントで必ずcamelCaseに変換してください。

5. setQueryData のqueryKeyが完全一致しているか確認してください。
保存後に setQueryData をしていても、実際に画面が読んでいる useQuery の queryKey と違っていたら意味がありません。

以下を確認してください。
- useGetEmployeeAllowances の queryKey
- getGetEmployeeAllowancesQueryKey に渡している引数
- setQueryData で使っている queryKey
- employeeId / payrollId / month / year などの引数が完全一致しているか

queryKeyが違う場合、キャッシュ更新は効きません。

6. 再編集時に古い空キャッシュで初期化しないようにしてください。
ただし「初回ロードだけ初期化」のロジックが原因で、後から正しいGET結果が来ても無視されている可能性があります。

現在の問題は以下の可能性があります。

- パネルを開く
- 古い空キャッシュが先に返る
- initializedRef が true になる
- その後、サーバーから正しい手当データが返る
- しかし initializedRef が true なので無視される
- 結果、手当が消えたままになる

これを防ぐため、初期化条件を以下にしてください。

- employeeId / payrollId が変わったら initializedRef を false にする
- queryが loading / fetching 中の古い空配列では初期化しない
- サーバー取得が成功したデータ、または保存直後に正しく更新されたキャッシュだけで初期化する
- 空配列で初期化するのは「本当に保存済み手当が0件だと確認できた場合」のみ
- staleな空キャッシュで initializedRef を true にしない

7. useQuery側で initialData や placeholderData が [] になっていないか確認してください。
もし placeholderData: [] や initialData: [] がある場合、それで初期化フラグが立ってしまう可能性があります。
その場合は、placeholderDataの空配列では rows 初期化を行わないようにしてください。

8. 保存後に再編集する画面が、本当に同じemployeeId / payrollIdでGETしているか確認してください。
保存時と再編集時でキーがズレている可能性があります。

例：
保存時 payrollId = A
再編集時 payrollId = undefined
取得時 employeeIdだけで取得している
または
保存時 monthlyPayrollId
取得時 payrollDetailId
のようなズレがないか確認してください。

9. 手当マスター allowanceDefinitions の読み込み前に、Selectがvalueを空に戻していないか確認してください。
row.allowanceDefinitionId が存在するなら、options未読込でもvalueは保持してください。
definitionsが未読込だからといって allowanceDefinitionId を "" にしないでください。

10. Selectコンポーネントが空文字 value を受けた時に、onValueChange が走ってrowを空にしていないか確認してください。
Radix/shadcn Select周りで、再レンダリング時に空値が入って state を上書きしていないか確認してください。

11. 保存APIが「一度全削除して再insert」方式の場合、insert対象に allowance_definition_id が含まれているか確認してください。
delete → insert の insert payload にIDがないと、毎回IDが消えます。

12. DBの型とフロントの型を確認してください。
allowance_definition_id が uuid なのか number なのか text なのか確認し、Select valueと比較する時は String() で統一してください。

修正のゴール：
- 手当を選択した直後、row.allowanceDefinitionId に値が入っている
- 保存payloadに allowanceDefinitionId が入っている
- API request bodyに allowanceDefinitionId が届いている

- GET APIレスポンスに allowanceDefinitionId が返っている
- rows初期化後に allowanceDefinitionId が入っている
- Selectのvalueが String(row.allowanceDefinitionId) になっている
- SelectItemのvalueも String(def.id) になっている
- 再編集時に「手当を選択...」ではなく、保存済み手当名が表示される
