まだ手当が再編集時に消えます。

前回の報告で「DB が空です。保存した手当がDBに残っていません」とありました。
この時点で、staleTime: 60_000 に変更してキャッシュを60秒保持する修正では根本解決になっていません。

重要：
キャッシュに表示されることではなく、DBに保存され、ページ再読み込み後・時間経過後・別画面遷移後でも再編集時に復元されることが正しいゴールです。

今回やってほしいこと：
staleTime や setQueryData の調整ではなく、まず「手当がDBに永続保存されていない原因」を根本的に調査・修正してください。

絶対に確認してほしいこと：

1. 保存ボタンを押した時、手当保存APIが本当に呼ばれているか
2. 手当保存APIのHTTPステータスが200/201で成功しているか
3. 保存APIのrequest bodyに手当データが入っているか
4. DB insert/update/upsert の対象テーブルが正しいか
5. insert/update/upsert の実行結果に保存データが返っているか
6. 保存後にDBを直接selectして、手当データが残っているか
7. 再編集時のGET APIが、保存した同じテーブル・同じキーで取得しているか
8. 保存時と取得時で employee_id / payroll_id / payroll_detail_id / month / year などのキーがズレていないか
9. 保存直後に別処理で手当データをdeleteしていないか
10. 「明細を確定」や「明細保存」など別APIが、手当テーブルを空配列で上書きしていないか

前回の「DBが空」というログが最重要です。
そのため、以下の流れで検証してください。

検証手順：

A. 手当を1件追加する
例：
- allowanceDefinitionId: 無事故手当
- amount: 5000

B. 保存ボタンを押す

C. 保存APIの直前ログを確認
console.log("[FRONT_SAVE_ALLOWANCES_PAYLOAD]", payload);

D. API受信直後ログを確認
console.log("[API_SAVE_ALLOWANCES_BODY]", req.body);

E. DB保存直前ログを確認
console.log("[DB_SAVE_ALLOWANCES_INPUT]", insertOrUpdateData);

F. DB保存結果ログを確認
console.log("[DB_SAVE_ALLOWANCES_RESULT]", result);

G. 保存直後にDBを直接selectして確認
console.log("[DB_AFTER_SAVE_ALLOWANCES_SELECT]", savedRows);

この savedRows が空なら、DB保存処理が失敗しています。
この場合、キャッシュやReact Queryは一切関係ありません。
DB保存処理を修正してください。

H. ブラウザをリロードする

I. 再編集画面を開く

J. GET APIのDB selectログを確認
console.log("[DB_GET_ALLOWANCES_QUERY_PARAMS]", {
  employeeId,
  payrollId,
  payrollDetailId,
  year,
  month
});

console.log("[DB_GET_ALLOWANCES_RESULT]", result);

ここで result が空なら、
保存時と取得時の検索条件がズレています。
保存したテーブル・保存したキーと、取得しているテーブル・取得条件を一致させてください。

特に疑ってほしい原因：

原因1：
手当保存APIが呼ばれていない。
画面の「保存」は給与明細本体だけ保存していて、手当行の保存APIを呼んでいない可能性があります。

原因2：
手当保存APIは呼ばれているが、payloadのフィールド名がAPI側と一致していない。
例：
フロント：
allowanceDefinitionId

API側：
allowance_definition_id

この変換がなく、DBにはnullで入っている、またはinsert対象から除外されている可能性があります。

原因3：
保存時は payrollId で保存しているが、取得時は employeeId + month で取得している。
またはその逆。
保存キーと取得キーを完全一致させてください。

原因4：
保存処理が delete → insert 方式になっていて、insert対象が空になっている。
deleteだけ実行され、insertが失敗している可能性があります。

原因5：
保存後に別のAPIが呼ばれ、空の手当配列で上書きしている。
特に「給与明細保存」「明細確定」「従業員明細更新」などのAPIが、allowances: [] を送って手当テーブルを削除していないか確認してください。

原因6：
DBテーブルが複数ある。
例：
保存先：
employee_allowances

取得元：
payroll_allowances

または
保存先：
payroll_detail_allowances

取得元：
employee_allowances

このようなテーブル不一致があると、保存成功しても再編集で出ません。
保存APIとGET APIで同じテーブルを使っているか確認してください。

原因7：
テナントID、会社ID、employee_id、payroll_idなどの条件不足で、取得できていない。
保存時のレコードには存在するが、GET時のwhere条件に合っていない可能性があります。

今回の修正方針：

1. staleTimeを根本解決として扱わないでください。
staleTime: 60_000 は一時的にキャッシュで見えているだけで、DBに保存されていなければ意味がありません。

2. 保存後に必ずDB selectで確認してください。
保存API内で upsert 後に同じ条件でselectし、実際にDBに残っているかログで確認してください。

3. DBに残っていない場合は、insert/update/upsert処理を修正してください。

4. DBに残っているのにGETで取れない場合は、GET APIのwhere条件またはテーブルを修正してください。

5. GETで取れているのに画面で消える場合だけ、React state / Select / query cache を確認してください。

優先順位は以下です。

第一優先：
DBに保存されること

第二優先：
GET APIで保存済みデータが返ること

第三優先：
rowsに正しく初期化されること

第四優先：
Selectに正しく表示されること

完了条件：
以下をすべて満たしたら完了です。

1. 手当を追加して保存
2. 保存API後のDB selectで、手当レコードが存在する
3. ブラウザを完全リロード
4. 再編集画面を開く
5. GET APIのDB selectで、保存済み手当レコードが返る
6. Selectに「手当を選択...」ではなく、保存済み手当名が表示される
7. 金額も保持される
8. 60秒以上経っても消えない
9. 別従業員に切り替えて戻っても消えない
10. キャッシュを使わず、DBから復元できる

修正後の報告では、必ず以下を書いてください。

1. 保存APIはどのファイル・どの関数だったか
2. 保存先DBテーブル名
3. GET APIが読んでいるDBテーブル名
4. 保存時のキー
例：
employee_id
payroll_id
payroll_detail_id
year
month

5. 取得時のwhere条件
6. DBに保存された実際のレコード例
7. GET APIで返った実際のレスポンス例
8. フロントのrows初期化後の実データ例
9. staleTimeなし、またはページ完全リロード後でも復元できること

重要：
「自動テストでシートを閉じて再度開いたら表示されました」だけでは不十分です。
それはキャッシュで見えているだけの可能性があります。

必ず、
- ブラウザ完全リロード
- 60秒以上経過後
- DB直接確認
- GET APIレスポンス確認
まで行ってください。

もし前回のログ通りDBが空なら、今回の最終結論は「React QueryではなくDB保存処理の不具合」です。
DB保存処理を最優先で直してください。
